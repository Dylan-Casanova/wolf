# Device Pairing Redesign

## Goal

Replace the current manual device registration flow with a self-registration + user-claiming model. ESP devices register themselves on first boot, then wait for a user to claim them from the web UI. This eliminates the broken state where a device tries to connect to MQTT before the backend knows about it, removes manual token handling, and supports multiple devices per user.

## Context

Currently, an admin must create a device record in the web UI (entering the device_id, type, user, and name) before the ESP can provision. The user also enters the device_id into the ESP's captive portal. If either side is wrong or the admin hasn't created the record yet, the ESP enters a broken state. Tokens are generated by the admin and displayed once in the UI.

The new flow automates all of this: the ESP self-registers, gets its token automatically, and waits for a user to claim it from the web UI.

## Firmware Changes

### Hardcoded Constants

Each ESP is flashed with a unique device_id and its type baked into the firmware:

```cpp
#define WOLF_DEVICE_ID    "ESP8266-001"
#define WOLF_DEVICE_TYPE  "esp8266"
#define WOLF_DEVICE_NAME  "ESP8266-001"  // default name, user can rename after claiming
```

These are set per-device at flash time. The provisioning base URL remains configurable:

```cpp
#define WOLF_PROVISION_BASE "https://iopen.it.com"
```

### Captive Portal

The captive portal form is simplified — it only collects WiFi credentials:

- WiFi SSID (text input)
- WiFi Password (text input — plain text, no masking, so the user can see what they type)

The Device ID field is removed from the portal.

### Boot Flow

The ESP has four modes, determined by what's stored in EEPROM:

1. **Setup mode** — No WiFi config in EEPROM, or double-reset detected. Starts captive portal AP ("Wolf-Setup") at 192.168.4.1. User enters WiFi credentials. On save, restarts.

2. **Registration mode** — Has WiFi credentials but no token. Connects to WiFi, calls `POST /api/device/register` with `{device_id, type, name}`. Receives token in response. Stores token in EEPROM. Enters polling mode.

3. **Polling mode** — Has WiFi + token but not yet paired. Polls `GET /api/device/{device_id}/status` every 30 seconds with the token in the Authorization header. When the response includes `paired: true` and MQTT config, saves MQTT config to EEPROM and enters normal mode.

4. **Normal mode** — Fully configured. Connects to MQTT, subscribes to command topic, listens for commands. Same as current behavior.

### LED Feedback

| Mode | LED Pattern |
|------|------------|
| Setup (captive portal) | Fast blink |
| Registration / Polling (waiting to be claimed) | Slow blink |
| Normal (operational) | Solid |
| Error (WiFi lost, MQTT failed) | Fast blink |

### EEPROM Layout Changes

The EEPROM layout is simplified. The `deviceId` field is no longer stored (it's hardcoded). The `serverUrl` field is no longer stored (it's derived from `WOLF_PROVISION_BASE`). Fields stored:

- WiFi SSID
- WiFi Password
- MQTT Host (received after pairing)
- MQTT Port (received after pairing)
- Device Token (received during registration)
- Reset flag (for double-reset detection)

## Backend API Changes

### New Endpoint: `POST /api/device/register`

Called by the ESP on first boot after WiFi setup. No authentication required (the ESP has no credentials yet).

**Request body:**
```json
{
  "device_id": "ESP8266-001",
  "type": "esp8266",
  "name": "ESP8266-001"
}
```

**Behavior:**
- If `device_id` is new: creates a device record with `user_id = null`, generates a token, returns it.
- If `device_id` already exists: returns the existing token (idempotent — safe for reboots and re-registration).
- Validates device_id format against a pattern (e.g., must match `ESP32-XXX` or `ESP8266-XXX`).
- Rate-limited: 10 requests per minute per IP.

**Response (201 or 200):**
```json
{
  "device_id": "ESP8266-001",
  "token": "a4574848247dc07c3f01acab7a6362f4..."
}
```

### New Endpoint: `GET /api/device/{deviceId}/status`

Called by the ESP every 30 seconds while waiting to be claimed. Requires device token.

**Headers:**
```
Authorization: Bearer {token}
```

**Behavior:**
- Verifies token against the device record.
- Updates `last_seen_at` on each poll (serves as heartbeat).
- Returns pairing status and MQTT config if paired.

**Response (unclaimed):**
```json
{
  "paired": false
}
```

**Response (claimed):**
```json
{
  "paired": true,
  "mqtt_host": "iopen.it.com",
  "mqtt_port": 1883
}
```

### Removed Endpoint: `GET /api/device/{deviceId}/provision`

Replaced by the register + status endpoints above. Remove the `DeviceProvisionController`.

## Database Changes

### Devices Table

- `user_id` becomes **nullable** — unclaimed devices have `user_id = null`.
- Remove the **unique constraint on `user_id`** — a user can now have multiple devices.
- All other columns remain the same.

### Device States (derived, not a new column)

| State | Condition | Meaning |
|-------|-----------|---------|
| Unclaimed | `user_id = null` | Registered by ESP, waiting for a user to claim |
| Claimed | `user_id = {id}` | Linked to a user, fully operational |

## Frontend Changes

### New: Claim Device Page

Accessible to any logged-in user. Simple form:

- One text input: Device ID
- Submit button: "Claim Device"

**Behavior:**
- On submit, backend looks up the device_id.
- If not found: error "Device not found."
- If already claimed by another user: error "Device is already claimed."
- If already claimed by this user: error "You already own this device."
- If unclaimed: links device to user's account, redirects to dashboard.

### Modified: Admin Device Management (`/devices`)

- **Remove the Create Device form** — devices self-register now.
- **Remove token display** — tokens are handled automatically.
- Admin can view all devices (including unclaimed ones, shown with "Unclaimed" status).
- Admin can reassign a device to a different user.
- Admin can unclaim a device (sets `user_id` back to `null`).
- Admin can delete a device.

### Modified: Dashboard

- Currently assumes one device per user. Must support multiple.
- If user has multiple devices, show a device selector or list.
- Each device shows its own controls based on type (garage button for ESP8266, garage button + stream view for ESP32-CAM).

## Security

### Registration Endpoint (`POST /api/device/register`)

- Open (no auth) — the ESP has no credentials at this point.
- Rate-limited: 10 requests per minute per IP.
- Validates device_id format — rejects IDs that don't match expected patterns (e.g., `ESP8266-XXX`, `ESP32-XXX`).
- Idempotent — repeat calls with the same device_id return the same token without creating duplicates.

### Status Polling Endpoint (`GET /api/device/{deviceId}/status`)

- Requires valid device token in `Authorization: Bearer` header.
- Rejects requests with missing or invalid tokens.

### Claiming

- Any logged-in user can claim an unclaimed device if they know the device_id.
- A claimed device cannot be reclaimed by another user — admin must unclaim it first.
- Admin can reassign or unclaim any device.

## End-to-End Flow

```
1. Developer flashes ESP with hardcoded WOLF_DEVICE_ID and WOLF_DEVICE_TYPE
2. User powers on ESP
3. ESP starts captive portal (no WiFi config yet)
4. User connects to "Wolf-Setup" WiFi, enters WiFi SSID + password
5. ESP saves WiFi to EEPROM, restarts
6. ESP connects to WiFi
7. ESP calls POST /api/device/register → receives token
8. ESP stores token, starts polling GET /api/device/{id}/status every 30s (slow blink)
9. User logs into https://iopen.it.com
10. User goes to Claim Device page, enters device_id from hardware label
11. Backend links device to user
12. Next ESP poll returns paired: true + MQTT config
13. ESP saves MQTT config, connects to MQTT broker (solid LED)
14. Device is fully operational — responds to commands from dashboard
```
